# Kai Solution Server: REST Frontend and Rule Candidate Staging

## Release Signoff Checklist

- [ ] Enhancement is `implementable`
- [ ] Design details are appropriately documented from clear requirements
- [ ] Test plan is defined
- [ ] User-facing documentation is created

## Open Questions

1. Per-archetype RBAC for rule approval needs a concrete design before GA. Do reviewers approve rules *for* an archetype, or is rule visibility scoped *by* archetype, or both? The initial implementation ships without per-role enforcement (matching the existing solution-server posture), but this must be resolved before the feature is GA-ready.
2. The expected end state for accepted rules is promotion into the Hub's existing systems (analysis profiles, custom migration targets) or a versioned git repository. The specific promotion mechanism (manual export via `/archive`, automated sync to a git repo, direct Hub API integration) needs design. How much of this should be automated vs. manual?

## Summary

The original Kai Solution Server enhancement ([solution_server.md](./solution_server.md)) defined an MCP-only service that collects migration data from IDEs and serves hints and success metrics back through MCP-compatible tools. Two assumptions from that proposal have not held up well in practice, and addressing them is what this enhancement is for.

The first is that MCP is sufficient as the only interface. MCP is the right shape for live LLM tool calls, but anything that wants to ingest batches, browse, paginate, filter, or have a UI built on top of it is better served by a normal HTTP API.

The second is that hints attached to existing rules are the only artifact worth mining from a migration. Hints make sense for incidents - every incident is by definition tied to an existing rule, so an incident is a natural anchor for advice on top of that rule. But rules themselves are also worth mining: any time a successful migration touches code the analyzer didn't catch, that's a candidate for a new rule. Rules are just another type of data we can extract from the migration work humans are already doing, and the server should be able to stage and triage them alongside hints.

This enhancement adds a REST frontend at `/api/v1/`, mounted alongside the existing MCP server and backed by the same database, and extends the data model so that the server can act as a staging and triage layer for rule candidates alongside hints. Rule candidates from any producer land in the solution server, go through a curation workflow (validation, filtering, accept/reject), and once accepted are promoted into the Hub's existing rule management systems or a versioned git repository. It also introduces a `collections` primitive for grouping related data - an import batch, a review session, a set of related contributions from a single tool run. The MCP surface is unchanged; everything here is additive.

## Motivation

The solution server's original goal was to turn observed migration outcomes
into reusable knowledge. Two things have changed since that proposal was
written:

**Direct IDE feedback has proven to be a weak signal.** "Did the developer
accept this fix?" is noisy for reasons that have nothing to do with whether
the fix is good: developers accept-and-edit, they accept-and-revert, they
ignore the diff and edit by hand, they reject correct fixes because they
prefer their own style. Treating raw accept/reject as ground truth produces
unreliable metrics and unreliable hints.

The solution server needs to admit other sources of migration knowledge -
reviewed commits from completed migrations, rules
imported from external rulesets, rules generated by other tooling. The API
should not privilege any single producer; it should give all of them the
same ingestion, curation, and serving surface.

**Hints are an under-ambitious output.** The solution server already has
incidents, violation groupings, before/after files, and reasoning. With a
small extension to the data model it can also stage rule candidates -
analyzer-lsp rules contributed by any producer - and triage them for
promotion into the Hub's existing rule management or a versioned git
repository. Multiple automated producers (commit miners, AI-assisted
authoring tools, semver-based generators) may all be generating rule
candidates, and those candidates need a place to land before they've
been reviewed. A database-backed staging layer can offer structured
filtering by validation state, provenance, and quality metadata,
which makes triage more tractable than reviewing candidates one at a
time. The accepted set that comes out the other end is what gets
promoted into centralized rulesets. (See Alternatives for discussion
of git-based triage as another option.)

### Goals

- Provide a stable, documented HTTP interface (`/api/v1/` with OpenAPI) for
  every solution-server use case that is not an interactive LLM tool call.
- Support browsing and querying incidents, solutions, violations, and hints
  with pagination and filtering, suitable for UIs and offline analysis.
- Introduce collections as a grouping primitive so that per-batch,
  per-review, and per-source scoping work uniformly across incidents,
  solutions, hints, and (in this enhancement) rules.
- Provide ingestion endpoints (all `POST` endpoints accept lists)
  suitable for any non-IDE producer without privileging any single
  producer in the API shape.
- Extend the data model and API so that rule candidates are first-class
  objects with producer-supplied provenance metadata, validation state,
  and a curation lifecycle.
- Provide a structured triage workflow for rule candidates that scales
  to the volume expected from multiple automated producers, with
  filtering by validation state, provenance, and quality metadata.
- Provide endpoints to assemble rulesets from accepted rule candidates
  and export them for promotion into the Hub's existing rule management
  systems (analysis profiles, custom migration targets) or a versioned
  git repository.

### Non-Goals

- Replacing or deprecating the MCP surface. MCP remains the primary interface
  for live LLM consumption inside Kai and other MCP-compatible tools.
- Designing a generic rule-authoring UI. This proposal exposes the data and
  endpoints a UI could be built on top of, but UI design is out of scope.
- Cross-Hub rule sharing or a global rule marketplace. Rules served by a
  given solution server are scoped to that Hub, matching the tenancy model
  of the original enhancement.
- Replacing or forking `konveyor/rulesets`. This proposal complements that
  repo and treats it as one possible upstream source of "stock" rules; it
  does not propose to move the canonical rulesets into the solution server.
- Permanently replacing the Hub's rule management or the `konveyor/rulesets`
  repository as the home for curated rules. The solution server stages and
  triages candidates; their permanent home is the Hub's existing systems
  (analysis profiles, custom migration targets) or a versioned git
  repository.
- Designing or specifying any particular rule producer. Commit miners,
  importers from external rulesets, hand-authoring tools, and AI-assisted
  authoring tools are all valid producers and are all out of scope for
  this enhancement; they are mentioned only as use cases that the API
  needs to accommodate. Specific producers belong in their own
  enhancements.
- Building the review mechanism itself. This proposal puts curation APIs
  in place (accept, reject, validation state) but does not build a review
  UI or automated validation service. Review is expected to happen
  externally - a human reviewer using the API directly, or eventually
  through a Hub UI. The design of that review surface is a separate
  concern.
- Any change to encryption-at-rest or code obfuscation beyond what the
  original solution server enhancement already specifies. (Per-archetype
  RBAC for rule approval is tracked as an open question and required
  before GA, but not part of the initial implementation.)

## Proposal

### Overview

The solution server gains two things:

1. A REST frontend at `/api/v1/`, mounted alongside the existing MCP
   surface and backed by the same data. The REST surface is purely
   additive; MCP behavior is unchanged.
2. A rules-and-rulesets data model that provides the APIs needed to
   build a curation flow for rule candidates. Any producer can submit
   candidates; the server stores them with provenance and validation
   metadata and exposes filtering, accept/reject state transitions,
   and export endpoints. The actual review process is built on top of
   these APIs. Accepted rules are exported for promotion into the
   Hub's existing rule management systems or a versioned git
   repository.

Auth and tenancy are unchanged from the original solution server
enhancement: all access is mediated by the Konveyor Hub proxy, and each
solution server instance is scoped to a single Hub. Several efforts in
the Konveyor community are exploring automated rule generation from
migration history and other sources; this API is intended to be the
staging destination for that work, providing triage and curation before
accepted rules are promoted into the Hub or a git repository.

### REST API surface

The REST API is mounted at `/api/v1/` and grouped into the following
resource families. Endpoints listed here are illustrative - the canonical
contract is the generated OpenAPI document at `/api/v1/openapi.json`.

#### Incidents (`/api/v1/incidents`)

- `POST /` - create one or more incidents (accepts a list).
- `GET /` - list with filters: `collection_id`, `ruleset_name`, `violation_name`;
  `offset` / `limit` pagination.
- `GET /{incident_id}` - full incident with `code_snip` and `variables`.
- `DELETE /{incident_id}`

#### Solutions (`/api/v1/solutions`)

- `POST /` - create one or more solutions (accepts a list), each linking
  incident IDs to before/after file content with optional reasoning and
  `used_hint_ids`.
- `GET /` - list with filters: `collection_id`, `status`.
- `GET /{solution_id}` - full solution including before/after file content,
  linked incidents, and linked hints.
- `DELETE /{solution_id}`
- `POST /{solution_id}/accept` - record a per-file acceptance signal.
- `POST /{solution_id}/reject` - record a per-file rejection signal.

The accept/reject endpoints exist for compatibility with IDE-driven feedback,
but this enhancement explicitly de-emphasizes their use as primary
ground-truth signals (see Motivation).

#### Violations (`/api/v1/violations`)

- `GET /` - list violation types (the (ruleset, violation) groupings under
  which incidents, solutions, and hints are organized).
- `GET /{ruleset_name}/{violation_name}` - detail with linked incidents and
  hints.
- `GET /{ruleset_name}/{violation_name}/success-rate` - the existing success
  metric, exposed over HTTP.

#### Hints (`/api/v1/hints`)

- `POST /` - create one or more hints (accepts a list), each linked to
  a `ruleset_name` / `violation_name` pair with content and optional
  reasoning.
- `GET /` - list with `ruleset_name` / `violation_name` filters.
- `GET /best?ruleset_name=…&violation_name=…` - the same "best hint"
  selection logic the MCP `get_best_hint` tool uses.
- `GET /{hint_id}` - single hint with link counts.
- `DELETE /{hint_id}`

#### Collections (`/api/v1/collections`)

Over time the solution server will accumulate artifacts from many
sources: different tools, different repositories, different migration
efforts. Without a cross-artifact grouping mechanism, the only way to
answer "what came from where?" is to query metadata fields on each
artifact type independently and correlate the results client-side.

A collection is a named group that spans artifact types. A tool that
produces rules, hints, and incidents in one run can tag all of them
with a single `collection_id`, and a reviewer can later filter any
artifact list by that collection. Collections also carry their own
metadata - source repo, migration type, description - so they serve
as a browsable index of what has been contributed and by whom.

Endpoints:

- `POST /` - create with `name`, `description`, `source_repo`,
  `migration_type`, free-form `metadata`.
- `GET /` - paginated list.
- `GET /{collection_id}` - detail with counts per artifact type.
- `PATCH /{collection_id}` - partial update.
- `DELETE /{collection_id}`
- `POST /{collection_id}/items` - add solution_ids, incident_ids,
  hint_ids, and/or rule_candidate_ids.

Collections are not a substitute for tenancy or RBAC. They are an
organizational handle within the existing Hub-scoped tenant boundary.

### Rule candidate staging and curation

This is the major data-model addition in this enhancement. Today the server
stores incidents → solutions → hints. After this enhancement it also stages
rule candidates and provides a triage workflow for curating them before
promotion into the Hub's existing rule management or a git repository.

#### Data model additions

- Rule candidate - an analyzer-lsp rule contributed by some
  producer. It carries the same fields an analyzer-lsp rule already
  has (rule ID, description, category, effort, message, labels, the
  `when` condition, links) plus:
  - Curation metadata: an optional grouping collection, a relevance
    judgment with reasoning, a validation state, and a curation state
    (`pending` / `accepted` / `rejected` / `superseded`).
  - Test fixtures: optional analyzer-lsp test data (sample source
    files and expected incidents) that a producer can submit alongside
    the rule. These serve two purposes: they let a validator confirm
    the rule fires correctly, and they give a human reviewer concrete
    examples of what the rule matches. Producers that generate rules
    from real code changes are well positioned to supply these.
  - Free-form `metadata`: producers populate this with whatever
    provenance is meaningful to them - commit hash, source repo,
    upstream ruleset name, IDE session id, model id, anything. The
    server does not impose a schema on `metadata`, but top-level keys
    are queryable via filter parameters (stored as JSONB) so that
    curation workflows like "all rules from producer X" or "all rules
    from commit Y" work server-side.

- Ruleset - a named grouping of accepted rule candidates, with
  labels so it can be filtered the same way the
  existing konveyor rulesets are. A ruleset is primarily an
  assembly and export mechanism: it groups accepted candidates so
  they can be exported as analyzer-lsp YAML (for validation or
  preview) or as an archive (for promotion into a git repo or
  import into the Hub). Its contents reflect the current set of
  accepted member rules; if a constituent rule is later rejected
  or deleted, it drops out automatically.

#### REST endpoints

- `POST /api/v1/rules/` - create one or more rule candidates (accepts a
  list).
- `GET /api/v1/rules/` - list with filters: `collection_id`,
  `migration_relevance`, `validation_state`, `curation_state`,
  `source_label`, `target_label`, `ruleset_name`, and top-level
  `metadata` keys. Paginated.
- `GET /api/v1/rules/{rule_id}` - detail with producer metadata,
  validation notes, linked example incidents/solutions, and the full
  rule YAML.
- `PATCH /api/v1/rules/{rule_id}` - update curation state, message, labels.
- `POST /api/v1/rules/{rule_id}/accept` - promote to `accepted`.
- `POST /api/v1/rules/{rule_id}/reject` - mark `rejected` with optional
  reason.
- `DELETE /api/v1/rules/{rule_id}` - delete a rule candidate.

- `POST /api/v1/rulesets/` - create or assemble a ruleset from a list of
  accepted rule IDs (or from a query: "all accepted rules in collection X
  with target=quarkus").
- `GET /api/v1/rulesets/` - list.
- `GET /api/v1/rulesets/{ruleset_id}` - metadata + member rule IDs.
- `GET /api/v1/rulesets/{ruleset_id}/yaml` - render the ruleset as
  analyzer-lsp YAML. This is primarily useful for validation and
  preview (confirming that the assembled rules parse correctly and
  fire as expected). It always reflects the current accepted members.
  Direct consumption by kantra is possible but the primary promotion
  path is through the Hub or a git repository.
- `GET /api/v1/rulesets/{ruleset_id}/archive` - download the ruleset as
  a zip/tar archive suitable for committing into a git repo or
  importing into the Hub. This is the primary export mechanism for
  promotion: a reviewer (or an automated job) exports the accepted
  ruleset and pushes it into the permanent home. Promotion to git
  could be automated as a post-acceptance hook.

#### Rule lifecycle

The solution server provides the APIs required to build a curation flow,
not a curation flow itself. The endpoints below support the full
lifecycle from ingestion through promotion, but the actual review
process - who reviews, what criteria they apply, whether it's manual or
automated - is built on top of these APIs, not prescribed by them.

```
   any producer
        │
        ▼
   POST /api/v1/rules            (curation_state = pending)
        │
        ├──▶ optional: automated or manual validation
        │    (validation_state = passed | failed)
        │
        ▼
   reviewer accepts              (curation_state = accepted)
        │
        ▼
   POST /api/v1/rulesets         (assembly into an export group)
        │
        ▼
   GET /api/v1/rulesets/{id}/archive
        │
        ▼
   promotion into Hub (analysis profiles, custom migration
   targets) or a versioned git repository
```

Validation is not a gate. A reviewer can accept a rule that has never
been validated - at that point the judgment is on them. Validation can
be performed by the producer at contribution time (a producer that has
access to source code can run `kantra test` against fixtures and post
the result alongside the rule), or by a separate service or human that
picks up unvalidated rules and updates them in place. The server stores
whatever validation outcome it is told; it does not enforce a
particular ordering between validation and acceptance.

### User Stories

#### Story 1: Migration knowledge needs a production home

As a developer building tooling that learns from migrations - whether
that's automated rule generation, hint extraction, or both - I need a
standard place to send what I produce. Today there is no such API.
The REST surface gives me documented endpoints that accept rule
candidates and hints with whatever provenance my tool tracks, and
feeds them into a curation pipeline I don't have to build myself.

#### Story 2: Building a curation UI

As a Hub UI developer, I want to build a screen where reviewers can
browse pending rule candidates and hints, inspect their provenance
and validation state, and accept or reject them. The REST API
provides the filterable list endpoints, curation state transitions,
and OpenAPI spec I need to generate a typed client and build this
without inventing a new backend.

#### Story 3: Promoting curated rules into existing systems

As a reviewer (or an automated job), I want to export the accepted
ruleset as an archive, push it into a git repository or import it
into the Hub's analysis profiles, and have kantra consume it through
the channels it already understands — without kantra needing to know
about the solution server.

#### Story 4: Building observability and debugging tools

As a developer building tooling around Konveyor, I want a documented
HTTP API I can use to trace data lineage in the solution server -
from hints back to the violations they improve, from rules back to
the incidents and solutions that motivated them. The REST API makes
this possible without requiring every tool to speak MCP.

### Implementation Details / Notes / Constraints

- Validation state is producer-supplied data, not something the server
  computes. A future enhancement may add server-side validation.
- Collections generalize the `client_id` pattern currently used by MCP.
  Today each MCP session uses a `client_id` to scope its data, which is
  effectively a nameless, metadata-less collection. MCP tools should
  adopt `collection_id` so that IDE sessions, REST contributions, and
  any other source share the same organizational structure.
- The MCP surface is otherwise unchanged. Existing MCP tools continue
  to work; `collection_id` adoption can be phased in.

### Security, Risks, and Mitigations

This enhancement does not change the underlying security posture established
by the original solution server enhancement. All access continues to be
mediated by the Konveyor Hub proxy; there is no direct external access to
the REST surface; data remains encrypted at rest in PostgreSQL; tenancy is
per-Hub.

Risks specific to this enhancement:

- Rules are higher-impact than hints. A bad hint misleads one
  developer on one incident; a bad rule fires on every codebase that
  runs the served ruleset. Mitigation: rules default to `pending` and
  are not served from a ruleset until they have been explicitly
  accepted, with validation state available as an additional gate at
  assembly time. Producer-supplied provenance is preserved so a
  reviewer can always trace a rule back to whatever produced it.
- Curation has no per-role enforcement initially. Any authenticated
  Hub user can accept a rule, matching the existing solution-server
  posture. This is more impactful for rules than for hints. Per-archetype
  RBAC is required before GA (see Open Questions) but may not be part of the
  initial implementation.
- OpenAPI exposes the data shape publicly inside the Hub. This is
  the intended consequence of having a documented HTTP API. The
  underlying data is already accessible to any Hub user via MCP; REST
  does not broaden the audience, it only makes the shape easier to
  discover.

## Design Details

### Test Plan

- Unit and integration coverage for the new REST endpoints, including
  the rule and ruleset lifecycle (contribute → validate → curate →
  assemble → export).
- An end-to-end test that contributes rule candidates, accepts a subset,
  assembles a ruleset, and confirms the served YAML is consumable without errors.
- Equivalence checks ensuring that any operation exposed by both MCP and
  REST behaves identically against the same data.

### Upgrade / Downgrade Strategy

The changes are additive: existing MCP clients see no behavior change,
and HTTP deployments simply gain a new `/api/v1/` surface after upgrade.
Downgrading disables the REST surface and the rules/curation
functionality without affecting existing MCP-mediated data.

## Drawbacks

- Two transports to maintain: Maintaining both MCP and REST means
  two surfaces to keep documented and consistent.
- Rules-as-data adds curation surface area: Storing rule candidates
  is easy; reviewing them is work. This enhancement provides the APIs
  for building a curation flow but does not solve the
  reviewer-bandwidth problem itself.
- The staging area needs a clear promotion path into existing systems
  (Hub analysis profiles, custom migration targets, or git repos).
  This enhancement provides the export endpoints (`/archive`, `/yaml`)
  but the specific promotion mechanism is not fully specified yet
  (see Open Questions).

## Alternatives

- Keep MCP as the only interface and extend it with bulk and rule
  tools: MCP is the right shape for live LLM tool calls ("given this
  incident, what hint should I show?") but the wrong shape for bulk
  writes (hundreds of rule candidates as individual tool invocations is
  awkward and lacks transactional semantics), browsing and pagination
  (shoehorning offset/limit/filters into tool arguments), curation
  workflows (accept/reject/promote wants stable URLs and PATCH
  semantics), and integration with non-MCP consumers (UIs, dashboards,
  CI jobs should not need to speak MCP; OpenAPI gives client generation
  for free). Both surfaces will continue to coexist - MCP for the
  live-agent use case, REST for everything else.

- Use git-based workflows for rule candidate triage: Producers submit
  rule candidates as PRs to a git repository, and the PR review process
  serves as the curation mechanism. This has real advantages - git is
  well-understood, PRs already have review/approve/reject semantics,
  CI can validate rules, and the merge is the promotion step. The
  tradeoff is that git doesn't provide structured filtering by
  validation state, provenance, or other metadata out of the box, and
  at higher volumes (multiple automated producers generating many
  candidates) someone has to own each PR individually. A database-backed
  staging layer can make triage more tractable at volume through
  queryable metadata and bulk operations. That said, git-based triage
  may be sufficient depending on the actual volume and number of
  producers, and the two approaches are not mutually exclusive - the
  solution server's `/archive` export could feed into a git-based
  review process rather than replacing it.

## Infrastructure Needed

No new infrastructure beyond what the original solution server
enhancement already specifies (PostgreSQL, in-namespace Deployment,
Hub-proxy authentication).
