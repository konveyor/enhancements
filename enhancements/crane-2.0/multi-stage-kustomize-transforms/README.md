# Inspectable Multi-Stage Transformations with Standard Kustomize Tools

## Release Signoff Checklist

- [x] Enhancement is `implementable`
- [x] Design details are appropriately documented from clear requirements
- [x] Test plan is defined
- [ ] User-facing documentation is created

## Open Questions

1. **Migration descriptor file vs. CLI flags + conventions (dev preferred):** Should Crane require a declarative descriptor file that specifies which transformers run and in what order, or is the current approach sufficient — CLI commands with flags (`--stage`) combined with the convention-based directory structure (`10_PluginName/`, `20_PluginName/`, `50_CustomChange/`)? The current approach keeps configuration minimal and co-located with the output. A future migration pipeline abstraction may introduce a migration plan file (including these descriptors) naturally, but it is unclear whether the transform descriptor file is needed at the single-tool level.
2. **Confirm strategy on rewriting transform stages and apply.** Stages backed by a plugin (e.g. default KubernetesPlugin) are not expected to be edited manually by the user, so could be safely re-created. Custom stages (not ending with `Plugin`) are expected to be user-specific modifications, so they need `--force` with transform to be re-created. Those custom stages are not idempotent, that is different to previous crane implementation. Need to confirm this is OK, or it is easier to tell each transform step re-creation needs `--force` and only `crane apply` stays idempotent.
3. **Enable helm templating within Kustomize.** While Kustomize is the standard for patching manifests, manually writing complex, conditional custom transformations (e.g. using components) might be difficult. One workaround is using AI (like Claude Code) to write custom plugins per use case. However, a simpler approach might be to allow Kustomize to evaluate Helm templates directly via `kustomization.yaml`. Note, those template files will not be generated by crane, so it will require users having experience with helm.
4. Is it right to assume:
    1. that user-created scripts around the previous `transform-*.json` files are not supported anymore? (not affecting plugins)
    2. that `crane apply` can call `kubectl kustomize` internally (or `oc`)? The binary itself will not have kustomize evaluation logic embedded.

## Summary

Crane's previous transformation command produces per-resource JSONPatch files (`transform-*.json`) and applies them for all plugins together with an in-process applier.

This enhancement replaces the custom pipeline with a **Kustomize-native output layout** and introduces a **multi-stage transformation pipeline** where each plugin operates in an isolated, inspectable stage. The result is a workflow that integrates with `kubectl`, `kustomize`, GitOps systems out of the box — while requiring zero changes to existing Crane plugins.

## Motivation

### The Overarching Problem

Crane exists to help users migrate Kubernetes workloads between clusters, distributions, and lifecycle tools (e.g., moving from manual `kubectl apply` to ArgoCD). The core workflow is **export -> transform -> apply**: export manifests from a source cluster, transform them to fit the target environment, and apply them.

Transformations are the critical middle step. A Deployment exported from old clusters may carry cluster-specific annotations, reference deprecated API versions, or embed image references that need rewriting. Crane's plugin system handles these mutations automatically — but today the *output* of that system is a set of opaque JSONPatch files that focus on resource manifests cleanup and minor changes.

This creates several real problems:

* **No standard customization by users.** Teams that want to layer additional changes (e.g., injecting a sidecar, overriding resource limits) must do so outside the Crane workflow, losing the single-pipeline benefit.
* **No traceability for multiple plugins.** When multiple plugins run, their patches are merged into a single file per resource. If a transformation produces unexpected results, there is no way to determine which plugin was responsible without re-running plugins individually.

### Why Now

The earlier [kustomize-support](https://github.com/konveyor/enhancements/tree/master/enhancements/crane-2.0/kustomize-support) enhancement was rejected because it offered "no more than `kustomize init --autodetect`." Since then, the plugin system has matured, the need for traceability has intensified (users run 3-5 plugins in production pipelines), and a detailed specification for multi-stage execution has been drafted. This proposal addresses the deeper integration that the earlier enhancement deferred.

### Goals

1. **Kustomize-native output.** `crane transform` produces a valid Kustomize overlay (`kustomization.yaml` + patch files) instead of flat JSONPatch files. Users can run `kubectl kustomize <transform-dir>` to preview the final manifests at any time.
2. **Multi-stage pipeline.** Each plugin produces its own isolated stage directory (`10_KubernetesPlugin/`, `20_OpenShiftPlugin/`, etc.). Each stage is independently inspectable and runnable. Each stage is suited for actual input manifests (from export or previous stage output), this means a new stage will internally apply the previous one.
3. **Debug-friendly stages.** Each stage (plugin and custom) produces its own input and output manifests under directory `transform/.work/10_ThePlugin`.
4. **Deterministic, Git-friendly output.** Patch files use stable naming (`<namespace>--<group>-<version>--<Kind>--<name>.patch.yaml`) and deterministic ordering so that repeated runs produce identical output when input is unchanged.
5. **Zero plugin changes.** Existing plugins continue to return JSONPatch operations. Crane handles the conversion to Kustomize patches transparently.
6. **Standard toolchain integration.** The output works with `kubectl`, `kustomize`, ArgoCD, Flux, OPA/Gatekeeper, and other tools that consume Kustomize overlays.

### Non-Goals

* **Replacing Kustomize.** Crane is not building its own overlay engine. It produces standard Kustomize artifacts and delegates rendering to `kubectl kustomize`.
* **Plugin distribution.** How plugins are discovered, installed, and updated is covered by the existing crane [plugin-management](https://github.com/konveyor/enhancements/tree/master/enhancements/crane-2.0/plugin-management) system.
* **Operator/controller integration.** This enhancement covers the CLI workflow only. Integration with the Crane operator or mig-controller is out of scope.
* **Semantic validation between stages.** Cross-stage content validation is delegated to `kubectl kustomize` at apply time.

## Proposal

### User Stories

#### Story 1: Transformation with the Default Plugin

**As a** platform engineer migrating an application, **I want to** run `crane transform` to execute the default `10_KubernetesPlugin`, **So that** server-managed metadata is removed, generating standard Kustomize patches that I can easily inspect and debug.

* **Patch Generation:** The plugin automatically generates JSONPatch operations that Crane transparently converts into Kustomize patches, saving them in `transform/10_KubernetesPlugin/patches/` using a predictable, Git-friendly naming convention (`<namespace>--<group>-<version>--<Kind>--<name>.patch.yaml`).
* **Inspection:** At any time, I can preview the final cleaned YAML before applying it to the cluster by simply running `kubectl kustomize transform/10_KubernetesPlugin/`.
* **Debugging:** If I need to see exactly what went into the transformation or what came out, I can check the intermediate raw snapshots inside the `transform/.work/10_KubernetesPlugin/input/` and `output/` directories.

#### Story 2: Multi-Stage Transformation with Multiple Existing Plugins

**As a** platform engineer performing a cross-platform migration (such as moving from OpenShift to vanilla Kubernetes), **I want to** run a multi-stage pipeline where each plugin has its own isolated directory (e.g., `10_KubernetesPlugin` and `20_OpenShiftPlugin`), **So that** I can accurately trace patch generation across plugins, inspect intermediate states, and debug unexpected changes without dealing with a single massive patch file.

* **Patch Generation:** Each plugin's output is isolated into its own stage directory with its own `kustomization.yaml` and `patches/` folder, meaning patches from different plugins are never merged into an opaque file.
* **Inspection:** I can inspect the output of any specific stage individually using standard tools like `kubectl kustomize <stage-dir>`.
* **Debugging:** If a resource (like a Route or Deployment) is incorrectly modified, I can easily isolate the behavior of a single plugin by inspecting its specific patch folder or by re-executing just that specific stage using the `crane transform --stage <stage-dir>` command.

#### Story 3: Custom Transformation Stage with User-Created Modifications

**As a** platform engineer adapting migrated resources to new environment standards, **I want to** append a custom transformation stage (e.g., `50_CustomModifications`) to the pipeline, **So that** I can layer GitOps-friendly declarative patches and overrides (like swapping image tags or updating namespaces) directly on top of the automated Crane output.

* **Modification Generation:** I can manually change the `kustomization.yaml` inside my custom stage directory and define custom modifications, like `namespace: migrated-app` or an `images:` block to swap container image tags, or changing annotation labels, or write custom patches without needing to write or compile a Go-based Crane plugin.
* **Inspection:** I can verify my custom changes alongside the automated ones by running `kubectl kustomize` on my custom stage directory, ensuring my edits correctly apply over the output of the previous stage.
* **Debugging:** If my manual edits to the `kustomization.yaml` are incorrect or malformed, running `crane apply` will fail and provide a reasonable error message for me to troubleshoot. If my modifications drift or I need to overwrite my dirty custom stage directory, I can force recreation using the `crane transform --force` flag. It is expected that the last stage is modified manually to be sure that the following stage will not be broken by the previous one.

### Implementation Details/Notes/Constraints

#### On-Disk Layout: Single-Stage

```
transform/
  10_KubernetesPlugin/
    kustomization.yaml
    resources/
    patches/
```

#### On-Disk Layout: Multiple-Stages

```
transform/
  10_KubernetesPlugin/
    kustomization.yaml
    resources/
    patches/
  20_OpenShiftPlugin/
    kustomization.yaml
    resources/
    patches/
  50_CustomModifications/
    kustomization.yaml
    resources/
```

Each stage consumes the `resources/` directory from the previous stage as input. The first stage consumes the export directory.

#### CLI Changes

**`crane transform` — new flags:**

| Flag | Purpose |
|------|---------|
| `--stage <dir>` | Transform only one stage |
| `--force` | Overwrite modified (dirty) stage directories |

**`crane apply` — behavioral changes:**

* Renders output with `kubectl kustomize`
* Discovers stages automatically from directory structure
* Supports same stage-selection flags as `transform`
* Preflight checks: validates `kubectl` availability and `kustomization.yaml` existence before execution
* Idempotent by design: `crane apply` delegates to `kubectl apply` with declarative semantics, so repeated runs produce the same result without side effects. Any future additions to the apply flow (pre/post hooks, logging, etc.) must preserve this property.

#### Plugin Compatibility

Existing plugins are unaffected. They continue to return JSONPatch operations via the existing plugin contract. Crane converts these operations into Kustomize Strategic Merge Patches or JSON6902 patch files during the serialization phase. This conversion is internal to `crane-lib`.

#### Stage Discovery

Stages are discovered by convention: Crane scans the transform directory for subdirectories matching `<number>_<pluginName>`. When no explicit configuration is provided, the KubernetesPlugin is assigned priority 10, and remaining plugins are assigned priorities starting at 20 in alphabetical order, with gaps of 10 to allow user-defined custom stages.

#### Determinism

For stable Git diffs and reproducible CI runs:

* Resources within type files preserve creation order
* `resources` and `patches` lists in `kustomization.yaml` are sorted lexicographically by filename
* Patch filenames follow the pattern `<namespace>--<group>-<version>--<Kind>--<name>.patch.yaml`
* Stage directories follow `<priority>_<pluginName>`

### Security, Risks, and Mitigations

**kubectl dependency.** `crane apply` gains a hard runtime dependency on `kubectl` or `oc`. Mitigation: preflight check with a clear error message; document the requirement.

**Kustomize rendering differences.** Kustomize may handle edge cases (e.g., `remove` on a missing path) differently than the current in-process applier. Mitigation: fixture-based parity tests comparing old and new output for the existing test suite.

**Stage state drift.** If a user modifies stage output manually and then re-runs `crane transform`, their changes could be overwritten, so `--force` option is required.

**Breaking change for existing users.** The `transform-*.json` file layout is removed. Mitigation: document the migration path; consider a one-release deprecation period where both formats are emitted.

## Design Details

### Implementation Epics

| Epic | Scope | Key Deliverables |
|------|-------|------------------|
| **A — crane-lib: Kustomize Foundations** | Library | `TransformArtifact`/`PatchTarget` types, Kustomize serializer, resource grouping, report structs |
| **B — crane: Transform Refactor** | CLI | Kustomize overlay generation, resource type files, deterministic ordering, dirty check |
| **C — Multi-Stage Pipeline** | CLI + Lib | Stage discovery, per-stage execution, stage-aware CLI flags, priority auto-assignment |
| **D — crane: Apply Refactor** | CLI | `kubectl kustomize` delegation, preflight checks, output handling |
| **E — Compatibility & Parity** | Testing | Plugin compatibility suite, edge-case regression tests |
| **F — Documentation & Rollout** | Docs | User migration guide, plugin-author notes |

### Test Plan

**Unit tests:**

* Kustomize serialization: JSONPatch to patch YAML conversion
* Resource grouping: correct multi-doc YAML generation
* Deterministic ordering: stable output across runs
* Stage discovery: correct parsing of `<number>_<pluginName>` directories

**Integration tests:**

* End-to-end `crane transform` -> `crane apply` -> diff against expected output for a reference set of exported manifests
* End-to-end with added custom stage with namespace change, add labels and changed image
* Plugin compatibility: run every built-in plugin against fixture data and verify output matches the previous JSONPatch-based steps, but is in the "kustomize" format.
* Multi-stage chaining: verify stage N correctly consumes stage N-1 output

**Edge-case regression tests:**

* `remove` operation on a path that doesn't exist causes failure
* Array index operations in patches
* Conflicting patches and whiteouts from multiple plugins - priority resolution by plugin stage order
* Empty/custom stage output (no patches generated, pass through-stage, no changes are added by default)

### Upgrade / Downgrade Strategy

**Upgrade:** Users running `crane transform` after upgrading will see the new Kustomize overlay output instead of `transform-*.json` files. Result manifests will still get into the output directory.

**Downgrade:** Users downgrading to a pre-enhancement version of Crane must delete their transform directory and re-run `crane transform`, as the old version cannot read the new layout.

## Drawbacks

* **Breaking change.** Removing `transform-*.json` files will break any user scripts or CI pipelines that parse the old format. This is mitigated by documentation and a deprecation period, but is still disruptive.
* **kubectl dependency.** Adding a hard runtime requirement on `kubectl` increases the tool's prerequisites. Users in air-gapped environments must ensure `kubectl` is available.
* **Complexity.** The multi-stage pipeline adds conceptual and implementation complexity. For users running a single plugin, stages add overhead without clear benefit. Mitigation: single-plugin runs produce a flat (non-staged) layout by default.

## Alternatives

1. **Embed a Kustomize library instead of shelling out to kubectl.** Pro: removes the `kubectl` runtime dependency. Con: Kustomize's Go library API is not stable and has historically been difficult to vendor. The `kubectl kustomize` path is simpler and always tracks the user's installed version.
2. **Keep JSONPatch files but add a `crane render` command.** Pro: no breaking change. Con: still requires users to learn Crane-specific tooling; does not integrate with the broader Kustomize/GitOps ecosystem.
3. **Use Helm charts instead of Kustomize overlays.** Pro: Helm provides great flexibility with templating system and it is widely adopted. Con: Typical k8s workload migration process aims to clean resources with minimal needed changes to make possible to be imported to the target cluster, templating should not be required in most cases. Helm's templating model is fundamentally different from patch-based transformation.
